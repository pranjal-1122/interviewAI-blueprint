01 — Core Concept

Not a question bank.
A performance intelligence system.

Most mock interview tools are glorified flashcard apps — they pick a question, you answer it, repeat. InterviewOS is different. It builds a live cognitive model of how you think, communicate, and perform under pressure, then adapts every question to target where you're weakest.

🔄

Adaptive Intelligence Loop

Every answer you give is analyzed for gaps. Weak on scalability? The AI generates a question targeting exactly that. Strong on architecture? It escalates. This is how real senior interviewers behave.

How Models Load — Automatically

You never touch a model file. Transformers.js fetches ONNX weights from Hugging Face's CDN the first time each pipeline runs, then stores them in the browser's IndexedDB cache. Every subsequent load is instant. Pre-cache all models on the demo laptop the night before your presentation. From that point, the app runs 100% offline.

03 — RAG System

Knowledge that teaches
the AI to ask smart questions.

The knowledge base never stores pre-written questions. It stores competency knowledge — what a great interviewer probes for, what signals a weak answer versus a strong one, and how to escalate. The LLM reads this knowledge and generates every question itself.

Why Not Fine-Tune?

Fine-tuning requires 10,000+ domain-specific training examples, GPU compute, and days of time. You have none of these. A small model fine-tuned on insufficient data performs worse than the base model with precise prompting. RAG gives you the same domain-specialization dynamically at runtime — and it's actually more architecturally impressive because it adapts to every candidate in real time. Knowing when not to fine-tune is a mark of AI maturity that judges notice.

What the knowledge base actually contains
// knowledge-base.json — competency knowledge, NOT questions { "backend_engineer": { "competencies": [ { "name": "System Design", "what_to_probe": "scalability thinking, tradeoff awareness, failure modes, cost", "weak_answer_signals": "only describes happy path, ignores scale, no numbers", "strong_answer_signals": "quantifies bottlenecks, mentions CAP theorem, discusses monitoring", "escalation_hint": "push toward distributed constraints if answer is strong" }, { "name": "Debugging & Ownership", "what_to_probe": "how they isolate problems, how deep they go, do they own the outcome", "weak_answer_signals": "vague, says 'I googled it', no reflection", "strong_answer_signals": "systematic approach, specific tools named, mentions prevention" } ], "answer_rubric": { "star_compliance": "Does the answer have Situation, Task, Action, Result?", "specificity": "Are real numbers, team sizes, and timelines mentioned?", "conciseness": "Is the answer scoped or does it ramble?", "role_relevance": "Does it align with the specific role's requirements?" } } }
RAG Pipeline — Runtime Flow
Step 1 — Parse
Role DNA Extraction
User describes the role. System extracts: role type, seniority, domain, required skills using the LLM.
→
Step 2 — Retrieve
Competency Lookup
Fetch the relevant competency objects, probe guidelines, signal definitions, and rubric from the knowledge base.
→
Step 3 — Augment
Context Injection
Retrieved competency knowledge injected into the system prompt. LLM now knows what to probe for, not what to ask.
Step 4 — Generate + Adapt
Dynamic Question Generation
After each answer, the LLM receives the full transcript, detects which competency you're weakest on, and generates a brand new question targeting that gap. No question is ever pre-written. No two interviews are the same.
→
Step 5 — Evaluate
Rubric-Based Scoring
LLM scores each answer 0–25 per dimension using the rubric from the knowledge base. Report generated as structured JSON.
System Prompt — Shaped by RAG, Not Fine-Tuning
const SYSTEM_PROMPT = ` You are Alex, a ${persona} conducting a real job interview. ROLE CONTEXT (from Role DNA Parser): Role: ${role} | Level: ${seniority} | Domain: ${domain} COMPETENCY KNOWLEDGE (retrieved from RAG): ${ragCompetencies.map(c => ` Competency: ${c.name} What to probe: ${c.what_to_probe} Weak signal: ${c.weak_answer_signals} Strong signal: ${c.strong_answer_signals} Escalation: ${c.escalation_hint} `).join('')} TRANSCRIPT + DETECTED GAPS SO FAR: ${transcriptSummary} YOUR RULES: - Ask ONE question at a time. Never two. Max 25 words. - Do NOT use any pre-written question. Generate one fresh using the competency knowledge above. - If weak signals detected → probe deeper on that competency - If strong signals detected → escalate using escalation_hint - Never reveal you are an AI - After each answer, score it: STAR(0-25) Specificity(0-25) Concise(0-25) Relevance(0-25) `;
Scoring Rubric
🎯

STAR Compliance

Did the answer include a Situation, Task, Action, and Result? Vague anecdotes with no outcome score low.

0–25 pts
🔍

Specificity

Were real numbers, team sizes, timelines, and concrete evidence used? Generic answers score low.

0–25 pts
✂️

Conciseness

Was the answer appropriately scoped? Rambling, repetition, and filler words reduce this score directly.

0–25 pts
🎯

Did the answer align with the specific role's competency requirements retrieved from the knowledge base?

0–25 pts

  • Answer comparison: yours vs AI-rewritten ideal
  • PDF report export via jsPDF

🚫 Cut for Hackathon

  • User authentication or any cloud persistence
  • Phi-3 or larger models — use Qwen2.5-0.5B
  • SpeechT5 TTS — browser synthesis is enough
  • Video recording and playback features
  • Facial expression or body language analysis
  • Vector database — plain JSON RAG works fine
  • Fine-tuning of any model whatsoever
  • Multi-round or panel interview simulation
  • Company-specific question databases
  • Real-time collaboration features
